Agent 智能体：会自己干活的 AI

感知 → 规划 → 行动 → 反思。当大模型装上循环和工具，它开始自主完成多步任务。

进阶 约 35 分钟

🎯 你将学会

✓一句话分清 Chatbot 和 Agent：一个"一问一答"，一个"给定目标，自主走完多步"—— 判断标准是球在谁手里
✓记住四件套：Agent = LLM（大脑）+ 工具（手脚）+ 循环（心跳）+ 记忆（工作台），每一件都来自前几课的旧零件
✓看懂循环的六拍，以及全课最重要的洞察：上一步的输出就是下一步的输入 —— 模型在自己的足迹上持续决策
✓诚实评估可靠性：明白错误为什么连乘累积，认得死循环 / 跑偏 / 成本爆炸三种死法，记住人机协作三件套解法

💡 核心概念：把"申请单回合"装进循环里

先盘点前四课攒齐的零件：第 16 课你学会指挥模型，第 17 课摸清它的记忆边界，第 18 课给它外挂资料，第 19 课它学会开工具申请单。但这些场景有个共同点：每个回合都由你发起，答完一轮，球就回到你手里。哪怕模型再强，这种"一问一答"的形态都叫 Chatbot。Agent（智能体）改变的正是这一点 —— 而它和你直觉里的样子，差了一个关键身位：

直觉印象
Agent 就是一个更聪明的聊天机器人

好像只要模型足够强，聊着聊着它就"升级"成了 agent。

真实机制
Agent 是把大模型装进循环里：给定目标后，它自主规划、调工具、看结果、再决策，干完才把球还给你

模型本身可以一模一样 —— 差别在外面那圈架构：循环、工具、记忆。

最贴切的画面是给实习生派活：你说"调研一下竞品，周五给我报告"—— 他不会每查完一个网页就跑来问"下一步干嘛"，而是自己列提纲、自己搜资料，链接挂了自己换关键词，实在卡死才来找你。
Chatbot 是"事事请示"的协作方式，Agent 是"给目标、交结果"的协作方式。注意：变化的未必是大脑的聪明程度 —— 是协作方式变了。

拆开任何一个 agent —— 无论产品包装得多炫 —— 里面都是同一副四件套，而且每一件你都在前面的课里见过：

大脑 · 第 10–14 课
LLM 大模型
唯一会"想"的零件。规划、选工具、读结果、做判断 —— 全是这台接龙机器在生成文本。
手脚 · 第 19 课
工具
搜索、读网页、跑代码、写文件。仍是"模型开申请单、宿主真执行"那一套，一字未改。
心跳 · 本课主角
循环
宿主程序里一个朴素的 while：任务没完成，就把最新结果喂回去、再跑一轮。"自主"全靠它续命。
工作台 · 第 17 课
记忆
上下文窗口摆着目标、计划和每轮足迹；长任务还要外挂笔记和摘要，防止工作台被堆满。

一张表把两种形态钉死 —— 顺便认认市面上的真实产品：

维度Chatbot · 一问一答Agent · 给定目标自主多步
交互方式 你发球它接球，答完一句回合即结束 你只给目标，它连续打完整局再交付
一次任务的步数 1 步：生成一段回答 几步到几百步：规划、调工具、看结果、再决策
出错的代价 错了你当场看见，下一句就能纠正 错误发生在中间步骤，会被后续步骤继承、放大（下文细讲）
典型产品 ChatGPT / Claude 的网页对话 Claude Code（自主写代码）、Deep Research（自主调研）、Manus 类通用 agent

提醒：agent 产品的格局变化极快 —— 表里点名的产品请当"形态代表"看，别当排行榜；具体能力以各家官网为准。

📖 循环拆解：六拍心跳，一轮接一轮

把"循环"放大到帧。每一轮心跳有六拍 —— 前两拍主要在开局走，后四拍周而复始：

1感知任务。读懂目标，盘点手头有哪些工具 —— 工具清单就躺在上下文里（第 19 课）。"目标"对模型而言只是窗口顶部的一段文字。记住这个事实，误区一会用到它。
2规划拆步。把大目标拆成子任务清单："先搜型号 → 再逐一查评测 → 最后汇总写建议"。计划本身就是接龙生成的一段文字（第 16 课思维链的近亲）—— 写出来后，它就成了后续每轮都会读到的路标。
3行动（调用工具）。从计划里取下一步，开一张工具申请单，宿主真正执行。第 19 课整套机制原封不动：模型从不亲手执行任何东西 —— agent 没有改变这条铁律，只是开单更频繁了。
4观察结果。执行结果回填上下文 —— 对模型来说，只是"窗里多了一段文字"。成功的结果是养料，失败的报错同样是养料 —— 演示里你会亲眼看到一次 404 如何变成转机。
5反思修正。符合预期吗？计划要不要改？这条路不通换哪条？"反思"听着玄，落地仍是生成文本：模型读着自己的足迹，续写一段自我评估。
6循环或退出。没完成 → 回到第 3 拍；完成 → 交付答案；反复碰壁 → 放弃并向人汇报。注意：判断"完没完成"也是模型生成的，它可能误判 —— 所以成熟产品常在这里设人工验收（下一节细讲）。

全课最重要的一句话：上一拍的输出，就是下一拍的输入。Agent 并没有长出新器官 —— 每一轮做决策的，仍是你在第 12–14 课认识的那台接龙机器：读上下文、续写下一段。唯一的变化是上下文里装的东西：不再只有你的问题，而是它自己一路留下的足迹 —— 计划、申请单、工具结果、报错、反思。模型在自己的足迹上持续决策。这句话立刻解锁两个推论：① 足迹越走越长，窗口越来越挤（第 17 课）—— 长任务必须做记忆管理（自动摘要、外挂笔记），否则开头的目标会被挤出窗外，agent 当场"忘了自己是谁"；② 足迹里一旦混进一个错误，后面每一轮都会把它当真去读 —— 这是下一节"错误连乘"的种子。

为什么非循环不可？因为多步任务的关键信息是"走到那一步才出现"的。调研耳机：不先搜一次，不知道该查哪三款；不点开链接，不知道页面早已 404。再聪明的模型也无法在第 0 秒写出一份永不需要修改的完美计划 —— 计划赶不上变化时，唯一的办法是走一步、看一步、改一步。单次生成（哪怕思维链写得再长）给不了"看一步"的机会，循环可以。

把你围观 agent 干活时见过的现象，和循环的节拍连上线：

你在产品里见过的现象循环里的哪一拍
Deep Research 一跑十几分钟，提示"已阅读 14 个来源" 循环跑到了第 N 轮 —— 每个来源都是一次完整的"行动 → 观察"
Claude Code 改完代码自己跑测试，测试红了又接着改 跑测试 = 行动；报错信息 = 观察；"我哪里改错了" = 反思 —— 一轮没过就再来一轮
屏幕上一行行滚过"正在思考… 正在调用工具…" 那不是装饰动画，是循环每一拍的实时日志 —— 产品把心跳播给你看
它跑到一半，你插一句"预算改成一千以内"，它真的会调整 你的话也成了上下文里的新足迹 —— 下一轮决策时被一并读到

📖 可靠性瓶颈：连乘的暴政

到这里 agent 听起来近乎完美：会规划、会动手、还会自我纠错。是时候泼冷水了 —— 这一节是全课最诚实的部分，也是你判断"agent 新闻是突破还是吹牛"的手感来源。

病根埋在第 14 课：模型的每一次输出都是概率采样，单步再准也只是"大概率对"。聊天里这无所谓 —— 错了你看得见，下一句就纠正。但 agent 把几十步串成一条链，每一步都踩在上一步的输出上，麻烦就来了：错误不是平均分摊的，而是连乘累积的。假设 agent 每走一步都有九成五的把握做对 —— 听起来已经相当可靠 —— 看看连续走下去，"全程一步不错"的把握还剩多少：

连续步数全程不出错的把握体感
1 步约 95%很稳
5 步约 77%开始心虚
10 步约 60%将将过半
20 步约 36%大概率中途已出错
50 步约 8%几乎必然翻车

这只是个示意模型 —— 真实任务里各步难度不同，有些错误还能被反思救回来。但量级感是对的：链路越长，"全程顺利"越接近抽奖。

错误累积之外，工程上还有三种常见死法 —— 每一种都在真实产品里反复上演：

死法一
死循环
同一个失败动作反复重试 —— 像游戏 NPC 卡在墙角原地踏步。烧着 token、产出为零；不设轮数上限，能一直卡到天亮。
死法二
跑偏
第 3 轮一个小误读（把"降噪耳机"看成"降噪音箱"），被后面十几轮当成既定事实继承 —— 越走越远，交付时已不知所云。
死法三
成本爆炸
每一轮都要把越滚越长的足迹整个重读一遍（第 17 课，按 token 计费）。轮数 × 足迹长度，账单飙升的速度远超直觉。

当前工程界的三件套解法 —— 共同思路只有一个：别让错误活过一轮，别让链条长到失控：

解法一 · 人把关
关键节点人工确认
业内叫 human-in-the-loop：删文件、花钱、对外发送 —— 这些动作必须停下来等人签字。第 19 课的安全铁律，在 agent 里因步数变多而加倍重要。
解法二 · 链条切短
子任务拆分
把 20 步长链切成几段 5 步短链，每段结束交付一个人能快速检查的小成果。连乘的链条越短，活下来的概率越高 —— 算术如此，没有魔法。
解法三 · 当场验货
可验证的中间产物
让每一步产出"机器或人能立刻核对"的东西：写代码就跑测试，做调研就留来源链接。错误当场拦截，而不是带病再跑十几轮。

这也顺手回答了一个现象级问题：为什么写代码的 agent 最先成熟？因为代码天生自带免费的验证器 —— 编译器和测试。每一轮"观察"拿到的都是客观硬信号，错误活不过一轮就被发现；改错了还能一键回滚。而"全自动炒股""全自动谈判"这类任务，反馈慢、噪声大、错误不可逆 —— 连乘衰减无人拦截。一条好用的经验法则：结果越容易被便宜地验证、错误越可逆的领域，agent 越早能用。下次看 agent 新品发布，先问这两个问题，比看 demo 视频靠谱。

⚠️ 常见误区

✗Agent 有自主意识 —— 它"想要"完成目标，失败了还"不甘心"地重试
✓"目标"只是你写进 prompt 的一段文字，"不达目的不罢休"是程序里写死的 while 循环

病因：围观 agent 连续干活十几分钟、碰壁后还会换路重试，实在太像"有意志"了 —— 这是人类把目的性投射到一切持续运动的东西上的老习惯。两个拆穿它的实验：① 把 prompt 里的目标换成任意别的字符串，它对新目标同样"执着"，毫无偏好；② 让宿主程序不再把结果回填回去（拔掉循环），"意志"当场消失，它退化成普通的一问一答。执着是架构的属性，不是心灵的属性 —— 欲望写在你的 prompt 里，毅力写在工程师的 while 里。

✗Agent 已经能全自动替代人类工作了
✓长链路成功率连乘衰减，目前最稳的形态是人机协作：AI 跑短链，人把关节点

病因：你刷到的 agent 演示视频，都是从无数次运行里挑出的成功案例剪的 —— 失败的那些不会出现在你的时间线上。真实工程里，长任务依旧会死循环、跑偏、烧预算，所以成熟产品全都保留着人工确认闸（Claude Code 每次删文件、跑命令都要先问你，就是明证）。当下最能打的用法不是"全自动"，而是把 agent 当一个不知疲倦的实习生：你定方向、切任务、验收中间产物 —— 它出手速，你出判断。

✍️ 小练习

1. 朋友说："我把聊天机器人接上了搜索工具，它现在算 agent 了吧？"用本课的四件套和"球在谁手里"标准，判断它还缺什么。
查看答案
对照四件套：有大脑（LLM）、有手脚（一个搜索工具）—— 但如果它仍是"你问一句、它搜一次、答完即止"，那就缺循环（自己决定下一步、连续跑多轮）和真正被用起来的记忆（在足迹上持续决策）。判断标准一句话：球在谁手里 —— 每一步都要你发起，它就还是"带工具的 Chatbot"；你只给目标、它自己走完全程，才是 agent。
2. 你让 agent 整理"近五年新能源车销量数据"，发现它跑到第 15 轮还在反复访问同一个打不开的网站。诊断这是哪种死法，并用本课的三件套解法各开一条整改方案。
查看答案
诊断：死循环 —— 同一个失败动作反复重试，token 在烧、产出为零（演示剧本里的 agent 第 4 轮就换了搜索词，这只 agent 的"反思"拍失灵了）。整改：① 人工确认：设轮数上限 + 连续失败若干次即暂停上报，让人来拍板换路；② 子任务拆分：把"近五年"切成按年份的五段短任务，每段单独跑、单独验收，一段卡死不拖垮全局；③ 可验证中间产物：要求每段先交付一张带来源链接的小表格 —— 数据是真是假、链接通不通，人和程序都能当场核对。
3. 为什么"AI 写代码"的 agent（Claude Code 这类）比"AI 全自动炒股"的 agent 先成熟？用"可验证的中间产物"和"错误连乘"两个概念解释。
查看答案
代码领域每一步都有便宜又客观的验证器：编译器、测试、报错信息 —— "观察"这一拍拿到的是硬信号，错误活不过一轮就被发现，相当于把每步成功率拉高、把连乘的链条切短；而且改错了能一键回滚（版本控制），错误可逆。炒股则相反：反馈慢（要等市场走完才知对错）、噪声大（涨了也未必是你判断对）、错误不可逆（亏掉的钱回不来）—— 连乘衰减一路无人拦截。经验法则：验证越便宜、错误越可逆的领域，agent 越早成熟。
