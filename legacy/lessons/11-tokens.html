Token：大模型眼中的世界

大模型不认识"字"，只认识 token。输入一句话，看它被切成什么样 —— 顺便搞懂为什么按 token 计费。

基础 交互演示 约 25 分钟

🎯 你将学会

✓一句话说清 token 是什么：文本的"乐高积木块"，每块对应一个编号
✓明白为什么必须切块：逐字看太碎、整词看会爆炸 —— token 是被两头逼出来的折中
✓亲眼看 BPE 怎么把高频组合一轮轮"焊"成新块：常见词整块、生僻词拆碎
✓用 token 视角解释一串怪现象：数不清字母、9.11 比 9.9 "大"、半个词往外蹦、按 token 计费
✓亲手玩一个示意版分词器，再猜猜各种词会被切成几块

💡 核心概念：先切块，再编号

大模型从不"看字"。你打的每句话，在进入模型前都会被切成一块块文本积木 —— token，每块对应词表里的一个编号。模型吃进去的是编号序列，吐出来的也是编号序列，最后一步才被还原成文字。

你以为
模型在逐字阅读「今天天气真好」
仿佛屏幕对面坐着一个识字的人，一笔一画都看在眼里。
实际上
它收到的是 [3742, 1102, 88, 451]
一串编号。哪个编号常和哪个编号搭配，它在训练中见得太多了；可"字长什么样"，它从未见过。

📖 一句话的旅程

从你按下回车，到屏幕上出现回答，文字其实经历了一趟"编号往返"。

① 你打的字
今天天气真好
→
② 切成 token 块
今天天气真好
→
③ 变成编号
[3742, 1102, 88, 451]
→
④ 模型计算
吐出新编号，再还原成文字给你看

负责"文字 ↔ 编号"来回翻译的程序叫分词器（tokenizer）。它不是模型的一部分，而是站在模型门口的翻译官 —— 编号是示意值，每家模型的词表和编号都不同。第 8 课讲过的 Embedding，正是把这些编号变成向量的下一步。

这趟旅程在"出口"那头同样成立：模型每算完一个新编号，系统就立刻把它翻译成文字发到你屏幕上 —— 这就是 ChatGPT、Claude 回答时字一小段一小段往外"蹦"的原因。蹦出来的最小单位是 token 块，不是字，所以偶尔会先蹦出半个词，下一拍才补全。

📖 深入展开 · 为什么非"切块"不可

看上去"切块"像是多此一举 —— 直接逐字读、或者按词读，不行吗？还真不行。token 是被两条死路逼出来的活路。

死路 ① · 逐字看（字符级）
句子变成超长队列
每个字一块，一篇千字文就是上千步。而模型内部"每块都要和每块打招呼"（第 10 课的注意力），队列一长，算力开销和工作记忆都被迅速撑爆 —— 像用米粒砌墙：什么都能砌，就是太慢太贵。
死路 ② · 整词看（词级）
词表收不完，新词不认识
网络新梗、人名、错别字、代码变量名……词是造不完的。凡是词表外的词，只能统统标成"不认识"，整段信息当场丢失 —— 像只用预制房间盖楼：图纸上没有的户型就盖不了。
活路 · 高频整块、低频拆碎
BPE：两头的好处都要
常见词焊成整块（省步数），生僻词拆到字节（任何输入都拼得出来，永远不会"不认识"），词表大小还能精确控制。这正是下一节的主角。

所以"切块"不是设计者的洁癖，而是工程上的最优折中。理解了这一点，BPE 的每一步都顺理成章 —— 它要做的，无非是自动找出"哪些组合值得焊成整块"。

🧱 BPE：把高频组合"焊"成积木

切块的方案不是人拍脑袋定的，而是从海量语料里统计出来的。主流做法叫 BPE（字节对编码），思想简单到一句话：谁总挨在一起，就把谁焊成一块。

第 0 步 · 全部拆碎
词表里只有几百个最小单元（字符或字节）：今天天气真好 —— 任何文字都能拼出来，只是切得很碎。
第 1 步 · 数频率
扫一遍语料：哪两块最常相邻？「今」+「天」可能出现了几十亿次，遥遥领先。
第 2 步 · 焊接成新块
把今天注册成一个新块，发一个新编号。从此它整块出场，不再拆开。
第 3 步 · 重复几万次
不断"数频率 → 焊接"，最终得到几万到二十几万块的词表（数量级，各家不同）。高频的词成了整块，低频的词只能用碎块拼。

光看文字不过瘾 —— 下面这个步进器，让你一轮一轮亲眼看焊接发生：同一句话从 17 块碎渣，怎么一步步缩成 9 块积木。

🎛️ 交互演示 · BPE 焊接步进器 点「下一步」逐轮焊接，红圈是本轮新块
这句话的 token 数

焊接顺序与频次为教学示意。真实 BPE 在字节层面操作、按全部语料统计频率，焊接会重复几万到几十万轮。

于是出现了一条铁律：越常见，块越大；越生僻，越碎。这直接决定了不同语言的"token 密度"——

英文 · 数量级
1 个 token 装下约四分之三个词
常见词整块（the、token），长词拆成子词（un·break·able）。1000 个 token 大约能装 750 个英文词。
中文 · 数量级
1 个字花 1 到 2 个 token
高频词可整词一块（"今天"），多数字单字一块，生僻字拆成字节碎块。不同分词器差异很大，记住数量级即可。
生僻字与 emoji
1 个字符花 2 到 3 个 token
词表里没有整块，就退回字节层拼凑 —— 一个 emoji 或冷僻汉字常要花两三块的"运费"。

那为什么不把块焊得更大、词表做得更猛，让一句话只占两三块？因为两头都有代价，BPE 的甜点区是试出来的 ——

块太大的代价
每块的"经验"变薄
块越长越专用，在语料里出现的次数就越少，模型攒不下足够的"语感"；而且几十万个块各自都要占一份参数，词表本身就会变得臃肿。
块太小的代价
每句话的队列变长
块太碎就退回死路①：序列变长、每一步都更贵、工作记忆吃紧。主流词表停在几万到二十几万块 —— 不是理论推出来的真理，而是工程上反复权衡的结果。

🔍 三个怪现象，一个解释

很多"大模型怎么连这都不会"的新闻，换上 token 眼镜一看，立刻不奇怪了。

怪现象 ①
数不清 strawberry 里有几个 r
它看到的不是 10 个字母，而是 strawberry 三个编号块。问它字母数，就像隔着电话问你"我刚才那句话一共多少笔画"。
怪现象 ②
认为 9.11 比 9.9 大
切块后是 9.11 对 9.9。逐块对照时「11」压过「9」—— 像版本号、像日期，就是不像小数。数字被切块后，比较并不天然按数值进行。
怪现象 ③
为什么按 token 计费
token 是模型每一步计算的基本单位：吃进多少块、吐出多少块，算力就花多少。所以 API 按 token 收费，模型的"工作记忆"上限（上下文窗口）也按 token 数 —— 第 17 课细讲。

📖 深入展开 · 现象反查：你在 ChatGPT / Claude 里见过的这些事

学会了机制，就能反过来"破案"。下面每一条你大概率亲眼见过 —— 左边是现象，右边是 token 层面的真相。

你看到的现象背后的 token 机制
回答一小段一小段地"蹦"，偶尔先蹦出半个词 模型一步只算出一个 token，系统算完立刻发给你（流式输出）。蹦出的最小单位是块不是字，半个词（一个子词块）先露面再正常不过。
同样的内容，中文对话比英文"烧"额度烧得快 中文 token 密度低：一个字常花 1 到 2 块，而英文一块能装大半个词。计费和上下文额度都按块数算，中文天然要交"语言税"。
让它倒着拼单词、数笔画、写藏头诗，常常翻车 字母和笔画在切块时就被"封进"块里了，模型看不见块的内部 —— 它靠训练时的统计印象在猜，不是在看。新一代模型常调用代码工具来绕过这个盲区。
聊得太长，它开始忘记你开头说过什么 模型的工作记忆（上下文窗口）按 token 数设上限，挤不下的部分会被截掉。也因此"长文档问答"贵且难 —— 第 17 课展开。
个别奇怪的字符串能让它当场胡言乱语 "故障 token"：编号在词表里、训练语料中却几乎没出现过，模型对这个编号毫无经验。早年著名的 SolidGoldMagikarp（一个论坛用户名）就是这样一块"幽灵积木"。

这套"现象 → token"的破案思路值得记住：以后再看到大模型犯莫名其妙的错，先问一句 ——"它看到的块，和我看到的字，是一回事吗？"多数谜团到这里就解开了。

🧩 猜猜切几块

检验一下手感：下面这些词在典型分词器里会被切成几块？先在心里报个数，再点卡片揭晓。块数因分词器而异，对上数量级就算赢。

⚠️ 常见误区

✗一个 token 就是一个单词 / 一个汉字
✓token 可能是半个词、一个词组、一个标点，甚至只是一个字节 —— 完全由语料里的出现频率决定

病因："词元"这个译名太像"词"了。其实 BPE 只认频率不认语法："今天"够高频就是一整块，"饕餮"太冷门就被拆成字节碎渣 —— 块的边界和人类的"词"边界经常对不上。

✗模型认识每一个汉字，理解它的字形和笔画
✓模型只认识 token 编号 —— "字"这个概念对它根本不存在

病因：拟人化想象。模型收到的永远是 [3742, 1102, …] 这样的编号序列，字长什么样、由几笔写成，它无从得知。它对"字"的一切感觉，都是从编号的搭配规律里间接学来的 —— 所以拆字、数笔画、玩字形谜语，恰恰是它的天然盲区。

✗分词器是模型的一部分，会在使用中跟着变聪明
✓分词器在模型训练开始前就定稿冻结了 —— 它只是一张查频率定下来的"切块对照表"

病因：把"翻译官"当成了"大脑"。流程上是先用语料统计出词表、冻结分词器，然后整个训练和之后的所有对话都用这同一张表。所以一个新梗火起来之后，老模型依然按旧表把它切成碎块 —— 这也是模型带"年代感"的原因之一。

✍️ 小练习

1. 朋友吐槽："连 strawberry 里有几个 r 都数不对，还叫人工智能？"请你用本课的知识替模型"喊冤"。
查看答案
模型看到的不是字母，而是 [str][aw][berry] 三个 token 编号。字母层面的信息在切块时就被"封装"了，它只能靠训练中的统计印象猜 —— 这是分词机制的盲区，不是智力问题。（新一代模型常靠调用代码工具来绕过这个盲区。）
2. 同样意思的一段话，用中文写和用英文写，哪个通常消耗更多 token？这对 API 账单意味着什么？
查看答案
通常中文更多：英文一个 token 平均装下大半个词，而中文一个字常要花 1 到 2 个 token，同义内容的中文 token 数往往更高。按 token 计费时，同样的话中文版常常更贵 —— 不过不同分词器对中文的优化差异很大，以实际切分为准。
3. 把 9.11 和 9.9 分别写成示意 token 序列，并解释模型为什么可能比错大小。
查看答案
9.11 → [9][.][11]，9.9 → [9][.][9]。模型不是按数值比较，而是按块的统计规律"接龙"：「11」压过「9」这件事在版本号、日期、章节号里大量成立，于是它顺势答错。把问题改写成"按小数比较"或让它列竖式，正确率会明显上升。
4. ChatGPT 回答时，字是一小段一小段"蹦"出来的，偶尔还会先蹦出半个词。用本课的机制解释这个现象。
查看答案
模型每一步只产出一个 token 编号，系统算完立刻把它还原成文字发给你 —— 这叫流式输出。蹦出来的最小单位是 token 块而不是字，所以半个词（一个子词块）先出现、下一拍再补全，完全符合机制。
